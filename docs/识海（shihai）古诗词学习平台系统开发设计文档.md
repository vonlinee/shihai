# 二、项目技术架构设计

## 2.1 核心技术栈选型

结合需求设计文档中的非功能需求（稳定性、易用性、可扩展性等），选用行业主流、高性能、易维护的技术栈，确保系统适配古诗词学习平台的业务场景，具体技术栈如下：

- 前端技术：React、Shadcn/UI、Tailwind CSS，实现页面美观、交互流畅，适配电脑、平板等不同终端，支持评论实时渲染、纠错进度实时更新等交互场景；

- 后端技术：Golang（Go）、Gin框架，高性能、轻量简洁，支持高并发访问，提升接口响应速度，适配多用户同时评论、纠错申请提交等场景；

- ORM框架：Gorm，简化数据库操作，实现数据层与业务层解耦，提升开发效率，便于后期数据维护与迭代；

- 数据库：PostgreSQL，关系型数据库，支持事务、复杂查询、高安全性，适合存储系统结构化业务数据（用户信息、诗词资源、评论、纠错申请等）；

- 辅助技术：
  
  - Redis：缓存热点数据（评论列表、点赞数、点踩数、待审核纠错申请计数），减少数据库查询压力，提升系统响应速度；
    
  - 敏感词过滤组件：对用户评论、纠错申请内容进行敏感词检测，保障平台内容合规；
    
  - IP哈希处理工具：对游客IP及浏览器指纹进行哈希摘要处理，生成唯一游客标识，保护游客隐私；
    
  - 消息推送组件：用于向资源审核角色、资源修改角色、纠错申请人推送相关通知（新纠错申请、投票结果、修改任务等）；
    
  - 密码加密组件：采用bcrypt加密算法，对用户密码进行加密存储，保障用户数据安全。

## 2.2 架构模式设计

系统采用“前后端分离 + B/S架构”模式，分层设计清晰，降低模块间耦合度，便于并行开发、独立部署与后期迭代，具体架构分层如下（从上层到下层）：

1. 前端层：负责页面渲染、用户交互，通过RESTful接口与后端进行数据通信，不直接操作数据库。主要实现用户界面展示、评论提交、纠错申请填写、投票操作等交互功能，适配不同终端的显示效果，确保用户体验一致。

2. 接口层：基于Gin框架实现，提供标准化RESTful接口，负责接收前端请求、进行参数校验、权限拦截、请求转发，将合法请求传递至业务逻辑层，同时将业务逻辑层的处理结果返回给前端，确保接口调用规范、安全。

3. 业务逻辑层：系统核心层，实现所有业务逻辑处理，对应需求文档中的全部功能，包括用户认证、诗词资源管理、评论互动（发布、点赞、点踩）、纠错申请处理、审核投票、资源修改、消息推送等，确保业务流程符合需求规范。

4. 数据访问层：通过Gorm框架实现，提供统一的数据访问接口，负责数据库的增删改查操作，将业务逻辑层的操作转化为数据库操作，同时处理数据库异常，确保数据操作安全、高效。

5. 数据存储层：负责系统所有业务数据的存储与管理，PostgreSQL存储核心业务数据（用户、诗词、评论、纠错申请等），Redis缓存热点数据，两者协同工作，提升系统性能与数据安全性。

## 2.3 架构优势说明

- 前后端分离：前端与后端独立开发、独立部署，前端可灵活迭代优化界面交互，后端可专注于业务逻辑与性能优化，提升开发效率，降低维护成本；

- 高并发适配：Gin框架+Redis缓存组合，支持同时在线1000+用户的并发访问，确保评论提交、点赞点踩、纠错申请等操作实时响应，无卡顿；

- 可扩展性强：模块独立设计，各层之间耦合度低，便于后期新增功能（如积分系统、打卡功能），无需大规模修改现有代码；

- 易维护性：技术栈主流、代码规范，分层清晰，便于开发人员排查问题、后期迭代，同时降低新人上手成本；

- 高安全性：多层权限拦截、密码加密存储、敏感词过滤、游客隐私保护等措施，全方位保障系统与用户数据安全，符合需求文档中的安全要求。

# 三、数据库详细设计

## 3.1 设计原则

结合需求设计文档中的数据需求与非功能需求，数据库设计遵循以下原则，确保数据安全、完整、高效、可扩展：

1. 数据完整性：通过主键、外键、唯一约束、非空约束等，确保数据的正确性与一致性，避免无效数据、重复数据入库，例如用户名唯一、纠错申请ID唯一等；

2. 高内聚低耦合：数据表按业务模块划分，每个表专注于存储某一类业务数据，表内数据关联性强，表与表之间通过外键关联，减少数据冗余，降低模块间依赖；

3. 高效性：合理设计索引，针对常用查询字段（用户名、诗词ID、评论ID、纠错申请ID等）建立索引，提升数据查询效率；优化字段类型，避免存储浪费；

4. 可扩展性：数据表结构预留扩展空间（如备注字段），字段类型选择兼顾当前需求与未来扩展，便于后期新增业务字段；

5. 规范性：数据表、字段命名遵循统一规范（英文小写，下划线分隔），字段注释清晰，便于开发、测试、运维人员理解与维护；

6. 安全性：敏感数据（如用户密码）加密存储，游客标识采用哈希处理，不存储原始IP与隐私信息，防止数据泄露。

## 3.2 核心实体与关系（E-R图核心关系）

对应需求设计文档中的角色与功能，系统核心实体包括：用户、朝代、诗人、诗词、诗词视频、诗词评论、评论点赞点踩、问答题目、答题记录、论坛帖子、论坛回复、公告、反馈、操作日志、纠错申请、纠错投票、消息通知等，各实体之间的核心关系如下，确保数据关联符合业务流程：

1. 用户 ↔ 评论：一对多关系，一个用户可发布多条评论，一条评论仅属于一个用户（游客无用户ID，关联visitor_id）；

2. 用户 ↔ 收藏：一对多关系，一个用户可收藏多条诗词，一条收藏记录仅属于一个用户；

3. 用户 ↔ 答题记录：一对多关系，一个用户可参与多次答题，一条答题记录仅属于一个用户；

4. 用户 ↔ 论坛帖子/回复：一对多关系，一个用户可发布多条帖子/回复，一条帖子/回复仅属于一个用户；

5. 用户 ↔ 反馈：一对多关系，一个用户可提交多条反馈，一条反馈仅属于一个用户；

6. 用户 ↔ 操作日志：一对多关系，一个管理员可产生多条操作日志，一条操作日志仅属于一个用户；

7. 用户 ↔ 纠错申请：一对多关系，一个用户可提交多条纠错申请，一条纠错申请仅属于一个用户；

8. 用户 ↔ 纠错投票：一对多关系，一个资源审核角色用户可对多条纠错申请投票，一条投票记录仅属于一个用户、对应一条纠错申请；

9. 用户 ↔ 消息通知：一对多关系，一个用户可接收多条消息通知，一条消息通知仅属于一个用户；

10. 诗词 ↔ 朝代/诗人/题材：多对一关系，多条诗词可属于同一个朝代、同一个诗人、同一个题材，一条诗词仅对应一个朝代、一个诗人、一个题材；

11. 诗词 ↔ 评论：一对多关系，一条诗词可有多条评论，一条评论仅属于一条诗词；

12. 诗词 ↔ 收藏：一对多关系，一条诗词可被多个用户收藏，一条收藏记录仅关联一条诗词；

13. 诗词 ↔ 诗词视频：一对多关系，一条诗词可关联多条视频，一条视频最多关联一条诗词；

14. 诗词 ↔ 纠错申请：一对多关系，一条诗词可有多条纠错申请，一条纠错申请仅属于一条诗词；

15. 评论 ↔ 子评论：一对多关系（自关联），一条评论可有多条子回复，一条子回复仅关联一条父评论；

16. 评论 ↔ 点赞点踩记录：一对多关系，一条评论可有多条点赞点踩记录，一条记录仅关联一条评论；

17. 问答题目 ↔ 答题记录：一对多关系，一道题目可有多条答题记录，一条答题记录仅关联一道题目；

18. 论坛帖子 ↔ 回复：一对多关系，一条帖子可有多条回复，一条回复仅属于一条帖子；

19. 纠错申请 ↔ 纠错投票：一对多关系，一条纠错申请可有多条投票记录，一条投票记录仅关联一条纠错申请；

20. 纠错申请 ↔ 消息通知：一对多关系，一条纠错申请可触发多条消息通知（推送至审核、修改角色及申请人）。
